01 — Self-Balancing Robot

Self-Balancing Robot

PID Control ESP32-S3 BNO08X IMU KiCad PCB L298N 3D Printed Chassis

A two-wheeled self-balancing robot built from scratch, including a custom PCB, 3D-printed chassis, and full firmware. The robot reads its tilt angle from a BNO08X IMU over I2C, computes a PID correction each loop, and drives an L298N dual H-bridge to keep itself upright.

Assembled robot — 3D-printed chassis (red PLA), L298N motor driver, BNO08X IMU, custom PCB, and 6×AA battery pack. The wiring harness connects the ESP32 to the IMU and both motor channels.

Mechanical Assembly

The chassis was designed in CAD and printed in PLA on a Prusa MINI at the Design-Innovation Building. Motors and wheels are secured with zip-ties for prototyping, then hot-glued once positions are finalized. The 3MF and BGCODE print files are included for anyone to reproduce the chassis.

Electronics and PCB

The PCB was designed in KiCad. It takes 5V and GND from the L298N motor driver's onboard regulator (no separate power supply needed), steps it down to 3.3V through an AP2112K LDO for the ESP32-S3, and breaks out I2C headers for the BNO08X IMU. The schematic follows the reference design in the project files with added decoupling capacitors placed close to the ESP32-S3 3.3V pin.

  • Microcontroller — ESP32-S3-Mini-1 with Wi-Fi and BLE, USB-C for flashing
  • IMU — BNO08X over I2C (SDA: GPIO 8, SCL: GPIO 9), using the ARVR stabilized rotation vector at 200 Hz for low-noise tilt readings
  • Motor driver — L298N dual H-bridge; ENA/ENB on PWM-capable GPIO 1 and 4, direction pins on GPIO 2, 3, 5, 6
  • Power — 6×AA (9V nominal) into L298N 12V input; L298N 5V output powers the PCB; AP2112K steps 5V down to 3.3V for the ESP32

Firmware and PID

On startup, the firmware averages 50 IMU readings to calibrate the upright setpoint, canceling out any hand wobble. Each loop it reads the roll angle (the forward/backward balancing axis for this IMU mounting), computes PID error relative to the setpoint, and drives both motors with the same signed correction.

P
Proportional
Kp = 20.0
Corrects current lean. Too high causes rapid oscillation.
I
Integral
Ki = 0.5
Eliminates persistent lean by accumulating past error over time. Anti-windup clamps the accumulator to ±50.
D
Derivative
Kd = 5.0
Predicts future error from rate of change, damping oscillation caused by P.

Tuning sequence: set Ki and Kd to zero, raise Kp until the robot oscillates rapidly, then cut it by about 40%. Next, raise Kd until oscillations damp without buzzing. Finally, add Ki to eliminate any persistent lean at rest. A 35-degree fallen-angle cutoff kills the motors and resets the integrator if the robot tips too far.

PID configuration snippet
// PID Tuning procedure (from the code comments):
// Step 1: Set Ki=0, Kd=0. Increase Kp until robot oscillates rapidly, then cut ~40%.
// Step 2: Increase Kd gradually until oscillations damp out.
// Step 3: Increase Ki slowly until any persistent lean disappears.

float Kp = 20.0;
float Ki =  0.5;
float Kd =  5.0;

// If a motor spins the wrong direction, flip between 1 and -1:

02 — Mini Aloha Robot

Mini Aloha

Robotics Raspberry Pi CAN Bus Servos Buck Converter Arduino Kinematics

The Mini Aloha is a compact bimanual robot platform (based on the AlohaMini open-source project). My contributions focused on the electrical integration and custom joystick control firmware.

What I Built

  • Power system — wired a buck converter to step down the supply voltage, and added a digital display showing live voltage and current draw across the system
  • CAN bus servo wiring — routed servo cables for both the drivebase and the arm joints, which communicate over CAN rather than standard PWM
  • Raspberry Pi integration — connected and configured a Raspberry Pi as the main compute board, plus a gyroscope for orientation sensing
  • Joystick controller firmware — wrote custom Arduino code to translate a 2-axis analog joystick into three-motor omnidirectional drive commands using trigonometric decomposition

Mini Aloha in action

The robot drives using three omnidirectional wheels, each powered by a CAN bus servo. The joystick firmware decomposes the stick vector into per-wheel power using trigonometric projection, giving smooth motion in any direction without stopping to reorient.

A buck converter steps down the supply voltage; a digital display shows live current and voltage draw. The Raspberry Pi handles higher-level coordination while the Arduino handles real-time joystick processing.

Omnidirectional Drive Math

The joystick maps to a 2D vector (x, y). The robot has three wheels spaced 120 degrees apart, with offset angles S1=30°, S2=150°, and S3=270°. The power sent to each wheel is:

Pn = (r / √2) × sin(θ − σn)

where r is the joystick magnitude, θ is the joystick angle (quadrant-correct arctangent), and σn is the offset angle of wheel n. This gives smooth, continuous omnidirectional motion with no abrupt transitions. Drift clipping zeroes out small stick deflections to prevent motor jitter at rest.

Joystick to omni-drive decomposition
// Arduino pin numbers
const int SW_pin = 13;
const int X_pin  = 34;
const int Y_pin  = 35;

const double pi = 3.14159265358979323846;

// Motor offset angles (radians) — positive spin = counterclockwise
const double S1 = 30  * (pi / 180.);
const double S2 = 150 * (pi / 180.);
const double S3 = 270 * (pi / 180.);

const double A_MAX = 4095.;  // max amplitude of joystick ADC
const int    M_MAX = 127;    // max amplitude of motors

// General Power formula:
//   Pn = r * sin(theta - sigma_n)
//   where theta is the joystick angle and sigma_n is the
//   positive offset angle of wheel n.  (See Will for derivations.)

void loop() {
  int x_raw = analogRead(X_pin);
  int y_raw = analogRead(Y_pin);

  double x_amp = (x_raw / (A_MAX/2.) - 1);
  double y_amp = (y_raw / (A_MAX/2.) - 1);

  // Drift clipping — zero out small stick movements
  double x = (x_raw > 2500 && x_raw < 2750) ? 0 : x_amp;
  double y = (y_raw > 2500 && y_raw < 2750) ? 0 : y_amp;

  double r0 = sqrt(pow(x,2) + pow(y,2));
  if (r0 != 0) { x /= r0; y /= r0; }

  // Quadrant-correct atan2
  double theta;
  if      (x == 0) theta = pi/2.;
  else if (x > 0)  theta = atan(y / x);
  else             theta = pi - atan(y / x);

  // Motor power in [-1, 1]
  double P1 = r0/sqrt(2) * sin(theta - S1);
  double P2 = r0/sqrt(2) * sin(theta - S2);
  double P3 = r0/sqrt(2) * sin(theta - S3);
}

03 — Lemon Run

Lemon Run

Arduino LCD Display Limit Switch Lemon Battery Game Dev

A side-scrolling video game running on an Arduino, displayed on a character LCD, and controlled with a limit switch — powered entirely by a lemon battery. The challenge was writing a playable game within the severe constraints of an Arduino's resources and the single-bit input of a limit switch.

  • Display — character LCD showing the game world as ASCII-art tiles, updated each frame
  • Input — a single limit switch (one button) for jump input
  • Power source — lemon electrochemical cell, demonstrating enough current to drive the Arduino and LCD
  • Platform — Arduino; game logic, collision detection, and scrolling all written in C++
Lemon Run — gameplay
Lemon Run — full demo

Powering a microcontroller from fruit is an exercise in understanding electrochemical cells: the lemon acts as an electrolyte between zinc (galvanized nail) and copper electrodes. Each cell produces around 0.9V; multiple cells in series provide enough voltage to run the system.

04 — Audio Electronics

LM386 Speaker Amplifier

KiCad PCB LM386 Analog Audio PCB Design

A compact audio amplifier PCB built around the LM386 low-voltage op-amp IC, designed in KiCad. The circuit takes an audio input through a volume-control potentiometer and drives a speaker with up to several hundred milliwatts, while a power LED and on/off switch round out the interface.

LM386 Amplifier Demo

The finished PCB driving a small 8Ω speaker from an audio source. The volume pot gives a smooth range from silent to full output. The Zobel network keeps the output stable across audio frequencies and prevents the high-frequency oscillation that an uncompensated LM386 can exhibit.

The board is compact enough to fit in a small enclosure, with all connectors (audio in, power, speaker out) accessible from the edges.

Circuit Design

The design follows the classic LM386 application circuit with several improvements. A 10kΩ potentiometer at the non-inverting input sets volume. A 100μF bootstrap capacitor (C2) between pins 1 and 8 increases the internal gain from 20 to 200. Output coupling through a 1000μF capacitor (C5) blocks DC from the speaker, and an RC Zobel network (R1 10Ω + C4 0.1μF) in parallel with the speaker load suppresses high-frequency oscillation. Bypass capacitors on the supply rail reduce noise and prevent motorboating.

Reference LM386 application circuit — audio input through 10kΩ volume pot, 100μF bootstrap cap on pins 1/8 for maximum gain, 1000μF output coupling cap, Zobel network (R1 + C4) on speaker output

KiCad Schematic and PCB

The schematic was captured in KiCad with a clean netlist covering all passive values, power flags, and connector footprints. The PCB was then laid out manually: the LM386 DIP-8 sits centrally, electrolytic capacitors are placed close to the supply and output nodes, and the power switch and LED are positioned along one edge for panel-mounting.

KiCad schematic — LM386, RV1 volume pot, SW1 power switch, D1 power LED, C2 bootstrap (100μF), C5 output coupling (1000μF), Zobel R1/C4
PCB layout — compact rectangular board; LM386 DIP-8 centered, large electrolytics on left and right edges, switch and LED front-row for easy panel access
